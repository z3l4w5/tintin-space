    2.1 你的第一个Hugo网站 #

    Hugo提供了一个广泛的命令行，显示了它的所有功能，包括引导一个新网站。 本节向你介绍Hugo的命令行。

    2.1.1 Hugo命令行 #

    Hugo是一个精心设计的命令行工具，提供了Hugo的所有功能。 它通过迁移数据、创建占位符、分析性能以及构建网站的核心任务来提供帮助。 Hugo命令行有两个不同的部分:

    • Commands —确定你想让Hugo做的任务。 你可以通过在命令行上使用hugo [命令] 来提供命令和子命令。 Hugo的命令是分层次的。 一个简单的hugo调用运行默认的command来构建网站。 发行Hugo new以创造新事物。 hugo new的默认创建新内容页面。 你可以使用hugo new site来构建一个网站骨架，并用hugo new theme来生成一个主题。
    • Flags (也称为命令行参数)—指定通过提供不同的配置来修改命令结果的选项。 标志是特定于命令的，每个命令可以有独立的标志。 例如, –format yaml在new site命令中将元数据格式从默认TOML更改为YAML。 学习Hugo命令行的直观方法是使用–help标志。 Hugo的帮助是分层的: Hugo–Help为Hugo命令提供帮助，并将Hugo new列为子命令； hugo new-help提供了新命令的文档，并将site作为子命令。 Hugo的帮助还显示了每个命令可用的所有标志。 你还可以生成手册页格式的Hugo命令行文档 (如基于UNIX的操作系统中的man命令所使用的)。 为此，请使用Hugo gen man或使用Hugo gen文档获取Markdown文件。 让我们通过创建我们的第一个网站来看看这一切是如何结合在一起的。 要在Hugo中创建一个新网站，我们将使用下面清单中的命令。
    代码清单2.1 Hugo命令创建一个新网站 
    hugo new site acme-corporation --format yaml

    此命令在当前文件夹中名为acme-corporation的子文件夹中以YAML为元数据语言创建Hugo skeleton文件夹结构。 此命令的各个部分如图2.2所示。 请注意，对于本书，我们将使用YAML ( https://yaml.org/) 代替默认的TOML (https:// toml.io) 元数据语言。 YAML在一般编程社区中更流行，没有TOML那么冗长，GitHub对它有更好的支持。 对于整个Hugo生态系统的新手来说，这是一种更容易入门的语言，也是更好的选择。 在第三章中，我们将讨论YAML作为Hugo的元数据语言。 附录b讨论TOML作为元数据语言选项。 请注意，官方的Hugo文档提供了所有受支持语言的元数据。

    2.1.2 添加到源代码控制 #

    任何项目的第一步都是将更改提交到版本控制存储库。 命令行界面没有本机撤消/重做支持。 如果你意外删除文件，该文件不会进入回收站或垃圾桶文件夹。 任何正在运行的脚本都有可能导致数据丢失，包括hugo命令。 除非你对源代码进行了版本控制，否则无法回头。 版本控制系统允许恢复已删除的文件并恢复到较旧的版本。 本书使用的版本控制系统是Git。 Git是最受欢迎的版本管理系统，GitHub对其集成紧密。 这也包括GitHub Pages，它是互联网上最受欢迎的静态网站主机。 将每个检查点提交到版本控制是一个好主意。 你可以使用git命令或像SourceTree或Fork这样的图形用户界面客户端来执行这些任务。 在命令行上，你可以使用多个Git命令执行此操作，如下面的清单所示。 为了帮助进行版本控制，请注意代码检查点，你可以在这些检查点暂停检查代码。

    2.1.3 Hugo源文件夹的结构 #

    在添加主题或某些内容之前，让我们先看看是什么组成了Hugo网站。 一个Hugo源文件夹不仅仅是模板和内容。 hugo新命令生成六个文件夹，我们将在使用Hugo的功能时创建更多文件夹。 如图2.3所示，我们网站中的关键文件夹包括：

    • archetypes —包含内容文件的模板。 Hugo试图最大限度地减少创建内容所需的复制和粘贴工作。 我们可以为此文件夹中的Markdown文件或文件夹创建模板，Hugo使用它们来创建基本内容文件。 我们将在第五章讨论原型。
    • content —包含传统上进入数据库的所有内容。 我们可以根据需要将内容组织到文件和文件夹中。 默认情况下，Hugo会根据此文件夹的结构直接生成网站输出，尽管我们可以使用每个文件中的元数据 (称为front matter，我们将在第3章中讨论) 覆盖该文件。 我们将在整本书中使用Content文件夹。
    • data —以YAML、TOML、CSV或JSON文件的形式存储结构化内容，这些文件作为全局变量在整个网站上可用。 传统的数据库存储的不仅仅是网页内容。 可以有与结构化数据相关联的表，这些表在内容文件夹中没有位置，因此当我们从Hugo外部生成内容并将该信息作为JSON或CSV文件传递给Hugo时，此文件夹派上用场。 我们将从第5章中的数据文件夹中读取数据。
    • layouts —覆盖主题的部分内容。 Hugo使我们可以灵活地混合和匹配主题页面，并编写自己的自定义页面。 在此文件夹中，可以进行主题的所有定制。 我们可以使用这个目录来存储这些被覆盖的主题布局。 主题和布局之间的界限是模糊的，Hugo给了我们完全的灵活性，通过一个接一个地覆盖页面来慢慢创建一个主题。 我们将在本章中使用layouts文件夹来更新主页，并在第6章和第7章中详细介绍layouts。
    • themes —包含我们用来使内容文件夹中的内容可呈现的代码。 我们可以使用Go模板语言来编写主题。 我们将在本章中添加主题，并在第7章中创建我们自己的主题。

    图2.3 Hugo中的网站源代码和内容位于源代码文件夹中。 Hugo new命令创建一组基本的文件夹，用于指定Hugo网站的各个部分：Archetypes文件夹(用于内容模板)、Content文件夹(用于文本内容)、Data文件夹(用于结构化内容和键-值对)、Layout和Themes文件夹(用于模板和单个页面设计)以及静态文件夹(用于需要托管但不适合任何其他类别的其他内容)。 在使用过程中显示的其他文件夹和文件包括资产 (用于未处理的图像和JS/CSS文件)，配置 (用于设置和元数据，最初作为单个文件生成)，资源 (用于缓存已处理的资产)，公共 (用于保存输出)，供应商和go.sum/mod (对于Hugo模块)，包 *.json和node_modules (对于JavaScript)，。github/netlify.toml (用于持续集成) 和api (用于自定义第一方api)。

    • config —包含该网站的配置。 此目录包含跨网站共享的元数据，包括主题的名称以及需要传递给Hugo或主题以呈现内容的任何参数。 默认情况下，Hugo创建一个config.yaml文件。 Hugo支持将此配置文件拆分为多个文件，并具有不同的测试和生产环境。 这会将配置转换为文件夹。 我们将在第四章中详细介绍该配置。
    • static—存储静态内容，如字体或pdf文件。 Hugo将此内容原样复制到输出目录。 这个文件夹有点等同于Apache/Nginx web服务器根文件夹，你可以在其中放置任何HTML文件进行渲染。 建议将尽可能多的内容放在Content、Data、Theme和Layout文件夹中，以便对其进行编程访问，并从Hugo的渲染管道中受益。 在静态文件夹中，我们可以存储二进制文件如pdf。woff (用于web字体)，和。压缩文件，用于不属于其他任何地方的可下载内容。 在本章中，我们将在静态文件夹中放置一些文件。

    在所有这些文件夹中，内容文件夹是我们通常花费最多时间向网站添加内容的地方。 Themes文件夹包含开发人员可以在网站外部管理的主题。 相比之下，我们不经常更改其他文件夹(数据驱动网页的数据文件夹除外)，只有在需要添加重要内容时才会更改。

    在建立基于Hugo的网站时，以下是你将遇到的其他一些文件和文件夹：

    • assets 文件夹—将图像、JavaScript和CSS文件作为未经处理的源代码放置在网站上供全局使用。 这个文件夹允许我们在编译期间处理这些文件。 Hugo可以调整图像大小，捆绑和缩小JavaScript文件，并通过其资源管道(Hugo Pipes)将SCSS转换为CSS。 We will learn about image manipulation and asset bundling in chapter 6 and work with JavaScript assets in chapter 10.
    • public 文件夹—Hugo的默认输出目录，其中hugo命令产生要在CDN上部署和缓存的HTML输出。
    • resources 文件夹 —处理数据时，Hugo会在此文件夹中缓存大量操作的结果。 我们应该将该文件夹放入我们的版本控制中，并跨版本重用它的数据。 这个文件夹是Hugo获得出色表现的关键要素之一。 处理图像是一项CPU密集型操作，需要花费时间。 大多数资产在构建过程中不会发生变化，只要处理过的图像不发生变化，缓存它们就可以显著提高Hugo的性能。
    • go.mod 和 go.sum 文件Hugo Modules使用这些文件来同步项目依赖关系。 我们很少查看这些文件，但我们确实需要将这些文件放在版本控制中。 We will introduce these files in chapter 8.
    • vendor 文件夹—存储我们可以通过Hugo模块包括的第三方依赖项。 我们将在第8章中使用Hugo模块时创建此文件夹。
    • node modules, package.json, package-lock.json, 和 package.hugo.json 文件—将Hugo与JavaScript生态系统联系并集成在一起。 我们将在第10章详细讨论在Hugo中使用JavaScript。
    • .github 文件夹和 netlify.toml 文件 —将Hugo与持续集成服务GitHub和Netlify联系起来。 我们将在整本书中使用这些服务。
    • api 文件夹—尽管不是标准配置，但我们将在第11章中创建此文件夹以容纳自定义api。
